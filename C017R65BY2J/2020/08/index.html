thinca 5日 23:47:18 Slack
実行させることはできた。 requested は workflow 着火時、 completed は完了時。 github.event.workflow_run.conclusion を見ると元の workflow の結果ステータスがわかる。 nil はまだ結果が出ていない(requested の時)、 'success' は成功、 'failure' は失敗、 'cancelled' はキャンセル
thinca 5日 23:55:03 Slack
元 workflow を着火したイベントが push の場合(それ以外は試してない)、 github.event.workflow_run.head_branch で push したブランチ or タグが取れる。ただし refs/heads/ や refs/tags/ は含まれておらず、タグ名かブランチ名かは区別ができなさそう。 github.ref は workflow_run 自体が着火した場所、つまりデフォルトブランチ refs/heads/master になっているのでこちらから元イベントの場所を知ることはできない
thinca 6日 00:24:02 Slack
当然だけど、job 側に if を設定して何も走らないようにしても、trigger の条件に引っかかれば workflow 自体は走っちゃう。「tag の push 時のテストだけ release する」ということがやりたくても、普段のブランチの push で空の release の workflow が走りまくることになる。
thinca 6日 00:25:37 Slack
そういえば if などによって job が 1 つも走らない workflow から workflow_run を起動した場合は、 conclusion は skipped になった
thinca 6日 00:46:06 Slack
あーそういうことか。意図した仕様かどうかはとても怪しいが、 workflow_run トリガーに渡せるオプションに branches があるのだけど、これはどうやら github.event.workflow_run.head_branch に当たる部分に適用されるらしい。つまり tag が push されたら tag の名前に対してマッチのチェックがされる
ということは branches: ['v*.*.*'] のように設定しておけば、バージョンタグへの push で走ったテストのみに workflow を走らせることができそう(job 側で conclusion のチェックも忘れずに)。当然、バージョンタグに似た名前のブランチを push してしまった場合にも走るので要注意。 v* くらいだと誤爆が起きる。 v2.0.0-dev みたいなブランチを作ると死ぬ
